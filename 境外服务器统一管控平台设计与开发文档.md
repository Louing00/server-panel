# 境外服务器统一管控平台：设计方案与开发文档

## 1. 项目目标

建设一个面向个人或小团队使用的 Web 平台，用来统一管理多台境外服务器。平台核心能力包括：

- 服务器资产管理：维护服务器名称、IP、端口、标签、分组、备注、登录方式等。
- Web SSH 终端：浏览器内直接打开服务器终端，支持多标签页、多会话、窗口 resize、断开重连提示。
- 文件管理：基于 SFTP 实现目录浏览、上传、下载、重命名、删除、新建文件夹、权限信息展示。
- 账号与权限：支持管理员、普通用户、只读用户等角色。
- 安全加固：凭据加密存储、操作审计、登录保护、敏感操作确认。
- 后续扩展：批量命令、主机监控、审计回放、堡垒机审批、OTP、WebDAV、反向代理等。

本项目第一阶段不建议做复杂 Agent。优先采用 agentless 模式：平台后端直接通过 SSH/SFTP 协议连接目标服务器，目标服务器无需安装客户端。

## 3. 总体架构

```text
Browser
  |
  | HTTPS / WebSocket
  v
Nginx / Caddy
  |
  v
App Server
  |-- REST API：登录、资产、用户、文件操作
  |-- WebSocket Gateway：SSH 终端实时通道
  |-- SFTP Service：文件浏览、上传、下载
  |-- Credential Service：凭据加解密
  |-- Audit Service：操作日志
  |
  | PostgreSQL：用户、服务器、凭据元数据、审计日志
  | Redis：会话、限流、临时任务状态
  |
  v
Remote Servers
  |-- SSH shell
  |-- SFTP
```

### 3.1 核心设计原则

1. 平台只做连接代理，不在服务器上安装 Agent。
2. SSH 凭据必须加密存储，不允许明文落库。
3. WebSocket 只作为浏览器与后端之间的实时通道，浏览器不直接接触目标服务器凭据。
4. 所有敏感操作都要记录审计日志。
5. 文件管理基于 SFTP，不走系统命令拼接，避免路径注入风险。
6. MVP 先保证单台连接稳定，再做批量管理。

## 5. 权限模型

### 5.1 角色

| 角色 | 权限 |
|---|---|
| admin | 全部权限，用户管理，系统设置，所有服务器 |
| operator | 可连接授权服务器，可上传/下载文件，可编辑自己有权限的服务器备注 |
| readonly | 只能查看服务器信息，可连接终端但不能操作文件写入，可选 |
| auditor | 只能查看审计日志 |

MVP 可以先实现 `admin` 和 `operator` 两个角色。

### 5.2 资源权限

权限粒度：

- 用户 -> 服务器组
- 用户 -> 单台服务器
- 用户 -> 操作能力

建议第二阶段再做细粒度权限。MVP 先用角色控制。

## 10. SSH 连接实现要点

### 10.1 后端流程

1. 校验 JWT。
2. 校验用户是否有访问该 serverId 的权限。
3. 读取服务器信息。
4. 读取并解密凭据。
5. 使用 ssh2 创建 SSH 连接。
6. 调用 `conn.shell()` 打开交互式 shell。
7. 将 shell 输出转发到 WebSocket。
8. 将 WebSocket 输入写入 shell。
9. 监听 resize 事件，调用 shell 的 setWindow。
10. 连接关闭时销毁 SSH 连接并记录审计日志。

### 10.2 伪代码

```ts
socket.on('connection', async (ws, request) => {
  const user = await authFromRequest(request);
  const server = await getServerAndCheckPermission(user, serverId);
  const credential = await decryptCredential(server.credentialId);

  const conn = new Client();

  conn.on('ready', () => {
    conn.shell({ term: 'xterm-256color', cols: 100, rows: 30 }, (err, stream) => {
      if (err) {
        ws.send(JSON.stringify({ type: 'error', message: err.message }));
        return;
      }

      stream.on('data', (chunk) => {
        ws.send(JSON.stringify({ type: 'output', data: chunk.toString('utf8') }));
      });

      ws.on('message', (raw) => {
        const msg = JSON.parse(raw.toString());
        if (msg.type === 'input') stream.write(msg.data);
        if (msg.type === 'resize') stream.setWindow(msg.rows, msg.cols, 0, 0);
        if (msg.type === 'close') conn.end();
      });
    });
  });

  conn.on('error', (err) => {
    ws.send(JSON.stringify({ type: 'error', message: err.message }));
  });

  conn.on('close', () => {
    ws.close();
  });

  conn.connect({
    host: server.host,
    port: server.port,
    username: server.username,
    ...credential
  });
});
```

## 11. SFTP 文件管理实现要点

### 11.1 连接策略

MVP 推荐每个文件操作单独创建 SFTP 连接，操作完成后关闭。实现简单，连接泄漏风险低。

第二阶段可加入连接池，但要注意：

- 同一用户、同一服务器可以复用短时连接。
- 空闲超过 60 秒关闭。
- 上传/下载大文件时不能被提前释放。

### 11.2 路径安全

必须处理：

- 禁止空路径导致误操作。
- 删除目录时需要二次确认。
- 默认不允许递归删除非空目录，后续再做。
- 路径必须经过 normalize。
- 不用 shell 命令执行 `rm`、`mv`、`mkdir`，必须调用 SFTP API。

### 11.3 下载流程

1. 校验用户权限。
2. 创建 SFTP 连接。
3. stat 目标路径，确认是文件。
4. 设置响应头。
5. 创建 readStream。
6. pipe 到 HTTP response。
7. 结束后关闭连接。
8. 记录下载审计日志。

### 11.4 上传流程

1. 校验上传大小。
2. 校验目标目录存在。
3. 使用临时文件名上传。
4. 上传完成后 rename 成正式文件名。
5. 记录审计日志。

## 14. 终端前端实现要点

1. 使用 xterm.js 初始化终端。
2. 使用 FitAddon 在容器 resize 时自适应。
3. WebSocket 收到 `output` 后写入 terminal。
4. terminal 的 `onData` 事件发送 `input` 消息。
5. terminal 的 resize 变化发送 `resize` 消息。
6. 断开后页面显示状态，不要悄悄失败。
7. 粘贴大段文本时做节流，避免一次性写爆后端。

伪代码：

```tsx
const terminal = new Terminal({
  cursorBlink: true,
  fontSize: 14,
  fontFamily: 'JetBrains Mono, Menlo, monospace',
  theme: {
    background: '#0b1020'
  }
});

terminal.onData((data) => {
  socket.send(JSON.stringify({ type: 'input', data }));
});

socket.onmessage = (event) => {
  const msg = JSON.parse(event.data);
  if (msg.type === 'output') terminal.write(msg.data);
};
```

## 15. 安全设计

## 15.1 凭据安全

- 数据库中绝不保存明文密码、私钥、passphrase。
- 使用服务端主密钥 `APP_MASTER_KEY` 对凭据进行 AES-256-GCM 加密。
- `APP_MASTER_KEY` 只能从环境变量或密钥管理服务读取。
- 支持 key_version，便于后续轮换主密钥。
- 日志中必须过滤 password、privateKey、passphrase、token。

## 15.2 登录安全

- 密码哈希：Argon2 或 bcrypt。
- 登录失败次数限制。
- 同 IP 限流。
- Access Token 短有效期。
- Refresh Token 可撤销。
- 管理员可以禁用用户。

## 15.3 SSH 安全

- 默认连接超时 10 秒。
- 默认终端空闲超时 30 分钟。
- 不允许用户在前端看到服务器凭据。
- 所有连接必须记录 userId、serverId、source IP。
- 可选：限制只允许连接数据库中登记过的服务器。

## 15.4 文件安全

- 上传文件大小限制。
- 下载文件要校验路径和文件类型。
- 删除目录必须二次确认。
- 默认不做递归删除。
- 不通过 shell 拼接路径执行命令。

## 15.5 审计

必须记录：

- 谁
- 在什么时候
- 从哪个 IP
- 对哪台服务器
- 做了什么操作
- 是否成功
- 失败原因

## 21. 风险与注意事项

### 21.1 最大风险：凭据泄露

解决：

- 加密存储。
- 日志脱敏。
- 严格权限。
- 生产环境主密钥不要写进代码仓库。

### 21.2 WebSocket 连接泄漏

解决：

- 断开浏览器连接时必须关闭 SSH。
- 设置空闲超时。
- 每个用户限制最大并发终端数量。

### 21.3 文件误删

解决：

- 删除前二次确认。
- 不默认递归删除目录。
- 审计日志记录完整路径。

### 21.4 平台自身暴露在公网

解决：

- 强制 HTTPS。
- 开启登录限流。
- 后续加 TOTP。
- 管理端可以加 IP 白名单。
- 定期备份数据库。

## 22. 推荐开发顺序

1. 先做登录和服务器 CRUD。
2. 再做连接测试。
3. 再做 Web SSH 终端。
4. 再做 SFTP 文件管理。
5. 最后补审计、设置、部署加固。

不要一开始就做批量命令和监控。SSH 终端和 SFTP 稳定后，这个平台就已经有实用价值。

## 24. 一句话总结

第一版做成“Web SSH + SFTP 文件管理 + 加密凭据 + 审计日志”的轻量控制台即可。它比完整堡垒机更容易开发、部署和维护，也更适合多台境外服务器的日常统一管理。
